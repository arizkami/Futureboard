# 999-AudioEngine.txt

Task: Design and implement the long-term Audio Engine + DSP architecture for Futureboard Studio using Rust WebAssembly for Web and a native .exe DSP service for Electron.

Goal:
Build the real audio engine architecture that will eventually replace the current WebAudio prototype.

This task defines the production-grade direction for:
- shared audio engine API
- Rust DSP core
- WebAssembly runtime for Web
- native DSP service for Electron
- IPC protocol
- audio graph
- mixer
- routing
- device/plugin chain
- built-in DSP effects
- transport/scheduling
- meter/telemetry
- offline rendering
- safe fallback paths
- low-error incremental implementation strategy

Important:
This is a long-term engine task.
Do not attempt to replace the entire current app in one giant unsafe patch.
Build the architecture incrementally with compatibility adapters.

Current app:
- React + Vite UI
- Electron shell
- WebAudio prototype
- project state / tracks / clips / mixer / inserts in TypeScript
- Future target: Rust/WASM for Web, native DSP service for Electron, later C++/SphereEngine rewrite

Main rule:
The UI must talk to one TypeScript AudioEngineAdapter API.
The UI must not care whether the runtime is:
- WebAudio
- Rust WASM
- Electron native service
- future SphereEngine

============================================================
0. Implementation Philosophy
============================================================

Build this in phases.

Do not break the current WebAudio prototype.

Required approach:
1. Define stable TypeScript audio engine API and protocol.
2. Implement a no-crash Rust DSP core skeleton.
3. Compile Rust core to WebAssembly for Web.
4. Wrap WASM in AudioWorklet for Web audio output.
5. Create native DSP service for Electron.
6. Communicate with Electron via safe IPC.
7. Keep WebAudio adapter as fallback until WASM/native are stable.
8. Add features in small testable layers.

Never do:
- direct UI calls into DSP internals
- hardcoded engine-specific logic in React components
- blocking audio thread with JSON parsing
- large memory allocations inside realtime process
- unbounded message spam from DSP to UI
- full plugin hosting before the internal graph is stable

3. Rust DSP Core Goals
============================================================

The Rust DSP core is the shared engine logic.

It should own:
- audio graph
- track mixer
- routing
- clip scheduling
- sample-accurate transport primitives
- built-in device DSP
- meters
- automation later
- offline render later

It should not own:
- UI state
- menus
- React state
- filesystem dialogs
- Electron APIs
- browser APIs

Core processing model:
- fixed block size or variable block size input
- non-interleaved f32 buffers internally
- sample rate aware
- no heap allocation in audio process after preparation
- no locks in realtime path where possible
- commands applied at safe sync points

Acceptance:
[ ] dsp-core builds as Rust library
[ ] Core can initialize with sample rate and block size
[ ] Core can process silent buffers without panic
[ ] Core has no UI/platform dependencies

17. Web AudioWorklet Design
============================================================

AudioWorklet architecture:

Main thread:
- WasmAudioEngineAdapter
- creates AudioContext
- audioContext.audioWorklet.addModule(...)
- creates AudioWorkletNode
- sends init command
- sends project/track/clip commands
- connects node to destination

Worklet thread:
- loads/receives WASM
- owns DSP engine
- receives commands through port.onmessage
- applies commands outside process or before next block
- process(inputs, outputs, parameters) calls DSP engine

Rules:
- process() must be deterministic and fast.
- Do not allocate big objects in process().
- Do not console.log in process().
- Avoid throwing exceptions inside process().
- Gracefully output silence on error.

Acceptance:
[ ] Worklet node can output silence
[ ] Commands can reach worklet
[ ] Transport start/stop commands accepted
[ ] No app crash on worklet init failure

21. IPC Performance Strategy
============================================================

Phase 1:
- stdio JSON-RPC for commands/status
- simple and debuggable

Phase 2:
- named pipe / Unix domain socket
- less overhead

Phase 3:
- shared memory ring buffers
- meter/telemetry and audio buffers

Never use JSON for:
- realtime audio buffers
- high-frequency sample data
- huge waveform data

JSON is okay for:
- project metadata
- transport commands
- parameter changes
- status/events

[ ] Phase 1 JSON-RPC works

- locks/mutex waits
- blocking IO
- JSON parsing
- logging
- panics
- dynamic graph rebuilds
- file decoding
- channel receives that block
- unbounded event creation

Allowed:
- preallocated buffers
- atomic param reads
- bounded command queues
- deterministic DSP operations
- fixed-size temp buffers

Acceptance:
[ ] process() path is reviewed for realtime-safety
[ ] Heavy work occurs outside process()
