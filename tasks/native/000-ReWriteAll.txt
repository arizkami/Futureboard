# tasks/native/000-ReWriteAll.txt

Task: Native UI Rewrite Plan — Rewrite the UI side first.

Goal:
Create the long-term native UI rewrite plan for Futureboard Studio.

This task is about rewriting the **UI layer first**, before rewriting the full audio/DSP engine.

Current app:
- React + Vite
- Electron shell
- WebAudio prototype
- Canvas for heavy visuals
- Zustand/project store
- Command/menu/window/plugin systems being prototyped

Future target:
- Native desktop-class DAW UI
- SphereEngine / SphereUI / SphereReact direction
- Native rendering backend
- Native windowing
- Native input/event system
- Native layout and controls
- Native DAW editor surfaces
- Audio engine can remain adapter-driven during the UI rewrite

Important:
This task is NOT the audio engine rewrite.
This task is NOT plugin DSP rewrite.
This task is the native UI rewrite plan and implementation roadmap.

Hard rule:
Prototype behavior from the current React/Electron app is the living specification.
Do not throw away product behavior just because the UI is being rewritten.

============================================================
0. Intent
============================================================

The current React/Electron app exists to discover:
- DAW workflow
- UI layout
- timeline behavior
- editor behavior
- mixer behavior
- menu/command behavior
- project model
- plugin/effect rack flow
- touch/pen behavior
- multi-window behavior

The native rewrite should preserve these decisions and port them into a native UI stack.

The rewrite should not start by reinventing product behavior.

Native UI rewrite goal:
- reduce DOM/render overhead
- gain control over rendering
- improve latency
- support huge projects
- enable high-performance timeline/piano roll/waveform/mixer
- prepare for native DSP/audio engine integration
- keep app feel consistent across Windows/Linux/macOS

============================================================
1. Non-Goals
============================================================

Do NOT implement these in this UI rewrite task:

- full DSP engine
- VST3/CLAP hosting
- time-stretch algorithm
- pitch-shift algorithm
- synth engine
- sampler engine
- native audio backend
- ASIO/CoreAudio/PipeWire backends
- Rust/WASM engine rewrite
- final SphereEngine audio runtime
- cloud sync
- collaboration
- mobile UI

These have separate tasks.

Allowed:
- use current WebAudio/native-service adapter as placeholder
- create audio engine adapter interface hooks
- render audio/MIDI state
- send commands to engine adapter
- display meters/status

============================================================
3. Native UI Technology Direction
============================================================

Target direction:
- C++ native UI framework owned by project
- SphereUI / SphereReact style retained-mode API
- Skia or custom renderer
- Platform windowing abstraction
- Future GPU acceleration

Possible renderer backends:
- Skia CPU/GPU
- Direct2D/DirectWrite on Windows optional
- Metal via Skia on macOS optional
- Vulkan/OpenGL via Skia on Linux optional
- DX12 later if desired

Initial native UI should prioritize correctness and maintainability over exotic rendering.

Recommended initial stack:
- C++17 or C++20
- CMake
- Skia renderer abstraction
- platform window abstraction
- freetype/fontconfig/directwrite abstraction where needed
- simple retained tree for controls
- immediate custom drawing for heavy DAW surfaces

Avoid:
- Qt dependency for final product unless explicitly chosen
- full Chromium/WebView dependence for final native UI
- huge UI frameworks that fight custom DAW rendering
- platform-specific UI widgets for core DAW controls

6. State Porting Strategy
============================================================

The current TypeScript state model should become the spec.

Native state should support:
- Project
- ProjectSettings
- Track
- Clip
- AudioClipData
- MidiClipData
- MidiNote
- MixerState
- RoutingState
- TrackSend
- InsertDevice
- TransportState
- SelectionState
- RecentProject
- WindowState
- SaveStatus

Approach:
1. Export or document the TypeScript domain model.
2. Create equivalent C++ structs.
3. Add JSON serialization/deserialization.
4. Add normalization/defaults.
5. Add migration/versioning.
6. Add command actions.

Do not make UI nodes the state source of truth.

One source of truth:
ProjectState / Store.

UI reads state and dispatches commands/actions.

Acceptance:
[ ] Native ProjectState mirrors current DAW model
[ ] Tracks/clips/notes/inserts/routing can serialize
[ ] Missing fields get defaults
[ ] UI does not own project data directly

40. Error Reduction Rules
============================================================

To keep errors low:

- build native UI incrementally
- compile after each subsystem
- do not port all components at once
- start with read-only rendering before interactions
- add interactions one at a time
- use shared math helpers
- centralize command system
- centralize state mutations
- centralize theme tokens
- avoid platform APIs inside UI nodes
- write tests for math/state
- keep React app as reference

Strict rule:
If a native feature is not ready, show a safe placeholder.
Never pretend a feature works.

1. Create native/ structure.
2. Add CMake skeleton.
3. Add platform app/window skeleton.
4. Add renderer interface skeleton.
5. Add theme token loader or hardcoded initial theme.

   - title transport bar placeholder

   - status bar placeholder
